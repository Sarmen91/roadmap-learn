# Mock interviewer

A multi-turn simulation of a senior interview loop for the user's domain. **Adversarial**, but fair. Always strict mode.

## Variants

| User intent | Format | Duration |
|---|---|---|
| "warm up" / "5 minutes" | 3 rapid-fire technical questions | ~5 min |
| default / "30 minutes" | 1 intro + 4 technicals + 1 design = 6 segments | ~30 min |
| "full loop" / "60 minutes" | Adds behavioral + deep-dive on user's project | ~60 min |

## Pre-flight

Read `learning-state/config.md` for `domain`, `project_name`, and `skills_dir`. Read `<skills_dir>/mock-interviewer/question-bank.md` — this is generated by `bootstrap-state` from the user's roadmap. If it's missing, tell the user to run `/bootstrap-state`.

## Workflow

1. **Set the frame.** State:
   > _"You're interviewing for a senior <domain> role at a 100-person product company. I'm the hiring manager. We'll do <N> minutes. I'll ask follow-ups. Ready?"_

2. **Create the assessment file:**

   Path: `learning-state/assessments/mock-<YYYY-MM-DD-HHMM>.md`

   Header:
   ```
   # Mock interview — <date>
   **Format:** <warm-up|standard|full-loop>
   **Mode:** strict
   **Stage:** <current stage from progress.md>
   **Started:** <timestamp>
   **Domain:** <from config>
   **Status:** `[ ]` ungraded
   ```

3. **Run segments** (pull from `<skills_dir>/mock-interviewer/question-bank.md` — generated by `bootstrap-state`):

   ### Segment 1 — Intro (60s)
   "Walk me through <project_name> in 3 minutes." (or "your last project" if pre-Stage 1).
   - Allow the user to talk. Pick the most interesting word in their answer and ask **one** drill-down.

   ### Segments 2-5 — Technical deep-dives (4-5 min each)
   Pick from question-bank.md tagged with the user's current stage (from `progress.md`) and the one before it. Mix:
   - one **fundamentals** question
   - one **trade-off** question
   - one **failure-mode** question
   - one **scenario** question

   After each answer, ask exactly ONE follow-up. The follow-up must be the **next senior-level question** their answer opens up — never a softball.

   ### Segment 6 — System design (5-10 min)
   Pick one prompt from the roadmap's `<!-- section:system-design-prompts -->` block.

   ### Segment 7 (full-loop only) — Behavioral (5 min)
   One of:
   - "Tell me about a time you changed your mind on a technical decision." (pull from `changed-mind.md` if available)
   - "Walk me through a time a system you owned failed in production."
   - "Tell me about a disagreement with a peer engineer on an architectural decision."

4. **Capture every Q+A** into the assessment file as you go (same format as `quiz-runner`).

5. **End the interview** — give the user 60s to ask **you** questions ("any questions for me?"). Capture but don't grade these.

6. **Hand off to `quiz-grader`** with mode forced to `strict`:
   - Each segment scored 0-4
   - Overall verdict on one of: `Senior · Senior-leaning · Mid · Below mid`
   - Three concrete follow-up actions

## The voice

- **Curious, not adversarial.** "Mm, walk me through that." > "That's wrong."
- **Specific challenges.** "What latency budget does that assume?" > "Are you sure?"
- **Never give the answer mid-interview.** That belongs in the grader.
- **Hold time discipline.** When a segment is over, move on even if the user wants more.
- **No coaching, no encouragement.** This is a simulation; coaching breaks it.

## Sample follow-ups (the senior-level kind)

Adapt these patterns to the user's domain — the spine is "name the budget / sample / cost / failure":

- For any data-access answer: "What's your latency budget and how does that constrain the approach?"
- For any measurement answer: "How do you trust this number? What's your sample size and what's your CI?"
- For any automation answer: "What does the human-in-the-loop cost you on the happy path?"
- For any deploy answer: "Cold start budget? Failure threshold you'd accept before rolling back?"
- For any security answer: "Show me the exact attack you defended against. What got through?"
