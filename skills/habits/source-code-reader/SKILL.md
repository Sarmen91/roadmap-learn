# Source code reader

Habit #4 from the roadmap. The fastest way to stop being intimidated by frameworks is to read 50 lines of their actual code.

## Suggested targets

Read from `.cursor/skills/source-code-reader/targets.md` — it's generated by `bootstrap-state` from the user's roadmap. The targets are mapped per stage. If the file is missing, tell the user: _"Run `/bootstrap-state` to seed targets, or pick a function yourself."_

The targets are starting points. Better targets are the ones the user is **currently fighting** — surface those first if the user has logged a relevant friction.

## Workflow

1. **Pick the target.** If user gave one, use it. Otherwise:
   - Read `targets.md` and the user's current stage from `learning-state/progress.md`.
   - Propose 2-3 from the table matching the user's current stage.

2. **Locate the source.** Ask the user to paste the function, OR ask them to clone the repo and provide the file path. The user runs the file fetch — this skill is read-only on their side.

3. **Set expectations:**

   > _"We'll read this in 3 passes: shape (30s) → mechanism (5 min) → trade-offs (5 min). I'll ask one question per pass. Ready?"_

4. **Pass 1 — Shape (30 seconds):**
   - User scans, doesn't read.
   - You ask: _"Without reading details: how many distinct things does this function do? Name the steps in plain words."_
   - Capture answer.

5. **Pass 2 — Mechanism (5 min):**
   - User reads paragraph-by-paragraph.
   - After each conceptual block, ask one comprehension question:
     - _"What invariant does this assert / check / preserve?"_
     - _"What happens if `<arg>` is None / empty / very large?"_
     - _"What's the error path here?"_
   - If user can't answer, point at the specific lines and ask again.

6. **Pass 3 — Trade-offs (5 min):**
   - Ask the senior-engineer questions:
     - _"What did the author choose to optimize for? What did they trade away?"_
     - _"What's the failure mode you'd test for if you depended on this?"_
     - _"How would you replace this function — could you, in your project?"_

7. **Log the read** to `learning-state/journal.md` as a sub-bullet of today's entry:

   ```
   ### Source-read — <library>.<function>
   Pass 1 verdict: <one-line>
   Mechanism aha: <what clicked>
   Trade-off learned: <one sentence>
   Time: <minutes>
   ```

8. **Seed 1-2 flashcards** for the strongest "aha" via `flashcard-maker`.

## Rules

- **Never paste large source blobs into chat.** Reference file + line ranges. Read 30-60 lines at a time max.
- **Never explain the function until the user has tried Pass 1.** Premature explanation defeats the drill.
- **Never go past 15 minutes total.** This is a habit, not a deep-dive.
- One read per session. Don't chain.
